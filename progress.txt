## Codebase Patterns
- Phase 1 search assertions: use `resultsIncludeSampleModule` in `src/tests/search/helpers.ts` because Fumadocs advanced search returns multiple fragments per URL and `attention` may rank glossary hits above the GQA page row.
- Search documents are built from `loadPublishedDocsPages(locale)` + `loadRegistry()` via `buildSearchDocuments` / `buildSearchDocumentsForLocale` in `src/lib/search/build-documents.ts`; body text and headings come from colocated `messages/<locale>.json`, not MDX prose.
- Registry aliases and tag record aliases merge into `SearchDocument.aliases` through `tagSearchTerms()`; page and registry `tags` populate `SearchDocument.tags` and facets.
- Static Orama snapshot: run `make build-search-index` (writes gitignored `src/generated/search-index.json`); Fumadocs advanced search uses `toAdvancedSearchIndexes` in `search-server.ts` at runtime without the artifact.
- Published docs pages need `page.mdx` frontmatter (`kind`, `registryId`, `status: published`) plus colocated messages; registry JSON under `src/content/registry/<kind>/`.
- Phase 1 sample content: GQA module at `src/content/docs/modules/grouped-query-attention`, token glossary at `src/content/docs/glossary/token`.
- Global search UI: `SearchDialog`/`SearchTrigger` register via `AppProviders` → `RootProvider` in `src/app/layout.tsx`; `DocsShell` renders `SearchTrigger` in the top nav for every `(site)` route; `HomeSearchEntry` opens the same dialog from `/`.
