# PRD: Phase 1 Fresh CI and Smoke Tests

## Introduction

Model Atlas Phase 1 discovery routes, search experience, and content baseline are largely implemented, but quality gates are not yet reliable from a **fresh checkout**. Standalone `make typecheck` fails because Fumadocs-generated `.source/` is gitignored and only produced today via `pretest` (before `bun test`), not before typecheck. There is no GitHub Actions workflow, README quality-gate docs are stale, and the architectural checklist requires CI on every pull request with deterministic Bun lockfile installs.

This work item hardens Phase 1 completion by making `make ci` pass after `bun install` alone, wiring baseline GitHub Actions that runs `make ci`, extending registry validation and route smoke tests with direct behavioral evidence, and resolving—or documenting—the Next.js/Turbopack filesystem tracing warning triggered by content loaders imported into doc pages.

**Depends on:** `phase-1-entry-and-index-routes` and `phase-1-search-experience-and-coverage` (Phase 1 routes, search surface, and discovery smoke baseline must exist).

**Enables:** `ideafy-loopback-phase-1-completion-batch-002` checklist sign-off and Phase 2 planning.

## Context

### Customer ask

Phase 1 completion and architectural-checklist hardening: make the quality gates reliable from a fresh checkout and wire baseline CI. Ensure Makefile/package scripts generate Fumadocs source before standalone typecheck when needed, keep `make ci` as lint, typecheck, test, build, and validate-data, and add a GitHub Actions workflow that installs with Bun's committed lockfile and runs `make ci`. Add or extend smoke tests that assert the Phase 1 route/content surface exists and that registry validation covers messages, assets, related docs, tags, and citations. Address the current Next/Turbopack filesystem tracing warning if it can be fixed by scoping content loaders cleanly; otherwise document the residual warning in progress or a technical note.

### Problem

A maintainer cloning the repo and running `bun install && make ci` can fail at typecheck because `src/lib/source.ts` imports `../../.source/server`, which is generated by `fumadocs-mdx` and excluded from git. CI is not automated in GitHub Actions, so pull requests lack enforced quality gates despite checklist requirements. Registry validation logic exists for tags, citations, messages, and assets, but negative coverage for related-document references and tag resolution is incomplete. Phase 1 route smoke tests and post-build static route verification exist but are not yet framed as the definitive CI content surface gate. `next build` emits a Turbopack NFT warning tracing filesystem-heavy content loaders through doc page imports.

### Solution

Add a package-script hook (for example `pretypecheck`) that runs `fumadocs-mdx` before `tsc --noEmit`, keep the `make ci` target ordered as lint → typecheck → test → build (with Phase 1 static route verification) → validate-data, add `.github/workflows/ci.yml` using `oven-sh/setup-bun` with `bun install --frozen-lockfile`, extend registry validation tests for unresolved `relatedIds` and tag references alongside existing messages/assets/citation cases, strengthen Phase 1 route/content smoke tests, and either scope runtime filesystem loaders away from Turbopack server-component import chains or record the residual warning in `progress.txt` with root cause and mitigation options.

## Description

Harden Phase 1 **quality gates and smoke coverage**: fresh-checkout `make ci`, baseline GitHub Actions, Fumadocs source generation before typecheck, expanded registry validation evidence, Phase 1 route/content smoke tests, and Turbopack tracing warning resolution or documented residual.

## Goals

- `make ci` succeeds on a clean worktree after only `bun install` (no manual `fumadocs-mdx` step).
- Preserve `make ci` as: lint, typecheck, test, build (+ Phase 1 static route check), validate-data.
- Add GitHub Actions CI on pull requests and pushes to `main` using the committed `bun.lock`.
- Prove registry validation rejects broken messages, assets, related-document IDs, tags, and citations with focused negative tests.
- Prove Phase 1 discovery routes, sample docs pages, and build output routes exist via smoke tests tied to real render/build behavior.
- Eliminate or document the Turbopack filesystem tracing warning from `next build`.

## High-Level Technical Design

### Fresh checkout CI flow

```txt
bun install --frozen-lockfile
        |
        v
make ci
  lint          -> biome check .
  typecheck     -> fumadocs-mdx (pretypecheck) -> tsc --noEmit
  test          -> fumadocs-mdx (pretest) -> bun test
  build         -> next build -> verify-phase-1-static-routes.ts
  validate-data -> validate-registry.ts
```

### Package ownership

- **Root Makefile / package.json** — canonical local and CI command surface.
- **`.github/workflows/ci.yml`** — remote enforcement mirror of `make ci`.
- **`src/lib/content/validate-registry.ts`** — registry and colocated page validation logic (extend tests, not duplicate parsers).
- **`src/tests/discovery/`** — Phase 1 route render smoke tests.
- **`scripts/verify-phase-1-static-routes.ts`** — post-build static route behavioral check.
- **`src/lib/content/*-pages.ts` and related loaders** — Turbopack tracing scope; keep filesystem IO out of modules imported synchronously by server components where possible, or narrow paths to `src/content`.

### Turbopack warning context (current)

`next build` reports an NFT warning tracing:

```txt
./src/lib/content/glossary-pages.ts
  -> ./src/lib/content/related-docs.ts
  -> ./src/features/docs/components/DerivedRelatedDocs.tsx
  -> grouped-query-attention page
```

Root cause: runtime `fs`/`path.join(process.cwd(), ...)` in content listing loaders pulled into the server-component graph. Mitigation options: lazy-load filesystem helpers inside route/data functions, split pure URL helpers from filesystem scanners, or add Turbopack-scoped path roots per Next guidance.

## Supporting Technical and UX Considerations

- `.source/` remains gitignored; generation must be deterministic and fast enough for CI (< few seconds).
- Do not broaden `make ci` with linkcheck or validate-pdf stubs—the customer explicitly scoped this batch to lint, typecheck, test, build, validate-data.
- Smoke tests should use static render (`renderToStaticMarkup`) or controlled subprocess checks—not brittle full browser E2E for Phase 1.
- GitHub Actions should cache Bun dependencies when practical but must not skip lockfile enforcement.
- Validation error messages must name record `id`, tag, or page path for reviewer-friendly CI logs.

## Success Metrics

- A new contributor runs `bun install && make ci` once and gets a green result without reading tribal knowledge about `fumadocs-mdx`.
- Pull requests show a passing CI check running the same commands as local `make ci`.
- Introducing a broken tag or citation reference fails `make validate-data` with a single actionable error in under five seconds.
- Phase 1 manual gate can rely on automated smoke and build route checks instead of ad hoc curl scripts.
